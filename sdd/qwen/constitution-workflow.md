# GitHub spec-kit 문서 생성 워크플로우

GitHub spec-kit 템플릿을 사용할 때 문서 생성의 추천 순서는 다음과 같습니다. 이 순서는 프로젝트가 명확한 사양에서 시작하여 계획, 실행, 검증 단계를 거쳐 최종적인 거버넌스 프레임워크로 나아가는 논리적 흐름을 따릅니다.

## 문서 생성 순서

1. **spec.md** - 사양 문서
   - 첫 번째로 작성
   - 프로젝트의 기본 사양과 요구사항을 정의
   - "무엇을" 만들 것인지 명확히 설명

2. **plan.md** - 계획 문서
   - 두 번째로 작성
   - spec.md를 기반으로 프로젝트 계획 수립
   - 일정, 리소스, 접근 방식 등을 설명
   - "어떻게" 그것을 만들 것인지 계획

3. **tasks.md** - 작업 문서
   - 세 번째로 작성
   - plan.md를 기반으로 구체적인 작업 항목 정의
   - 구현 단계를 위한 실질적인 작업 단위

4. **checklist.md** - 체크리스트 문서
   - 네 번째로 작성
   - 완료 기준과 검증 단계를 정의
   - 프로젝트 성공 여부를 판단하기 위한 기준

5. **constitution.md** - 헌법/구조 문서
   - 다섯 번째(마지막)으로 작성, 또는 가장 먼저 작성하기도 함
   - 다른 모든 문서가 완료된 후 작성
   - 프로젝트의 거버넌스 및 의사결정 프레임워크를 정의
   - 역할, 책임, 의사결정 과정 등을 규정


## constitution.md의 위치와 중요성

constitution.md는 워크플로우에서 가장 마지막에 작성되는 문서입니다. 그 이유는 다음과 같습니다:

- 다른 모든 핵심 문서(spec, plan, tasks, checklist)가 먼저 완성되어야 구성할 수 있음
- 이미 정의된 사양, 계획, 작업, 검증 기준을 기반으로 거버넌스 구조를 설정
- 프로젝트의 의사결정 프레임워크가 실제 내용물에 맞춰져야 하므로
- 팀의 역할과 책임이 구체적인 작업 내용을 반영해야 하므로

## 워크플로우 요약

```
spec.md → plan.md → tasks.md → checklist.md → constitution.md
  ↓         ↓         ↓          ↓             ↓
사양 정의   계획 수립  작업 정의   검증 기준     거버넌스
```

이러한 순서로 문서를 작성함으로써, 프로젝트는 체계적인 구조를 가지게 되며, 각 문서는 이전 단계의 산출물을 기반으로 다음 단계를 정의하게 됩니다. 특히 constitution.md는 프로젝트 전체를 통합적으로 관리하고 조정하는 최종 거버넌스 문서로서의 역할을 수행합니다.
